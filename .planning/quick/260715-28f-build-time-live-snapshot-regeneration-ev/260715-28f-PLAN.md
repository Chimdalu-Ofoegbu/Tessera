# Quick Task 260715-28f: Build always ships live data (never stale)

## Problem
Prod serves a static `public/api/*.json` snapshot. It was generated **once, by hand**
(the README told a human to curl the endpoints), so it froze at asOf Jul 7 while the live
Renaiss Index moved on (Jul 14: PKM 11,255→13,083, OPC 20,811→24,791, both indices flipped
from down-week to up-week, risk 8/11→7/9). The build never refreshed it.

## Constraint that shapes the design
Renaiss public tier = **10 requests/day per IP**. Per-request/per-visitor live fetching is
impossible (a few hundred page loads would exhaust it and 429 the site). So "always live"
must mean **refreshed at build time**, not at request time. Runtime-live would need partner
credentials (10k/day) — offered as a follow-up, not assumed.

## Solution — regenerate the snapshot on every production build, fail-safe
1. `scripts/gen-snapshot.mjs` — loads the real compute layer via Vite `ssrLoadModule`
   (same mechanism as the dev middleware), runs `buildOverview` / `buildCategoryAnalytics`
   / `source.health()` over the live `RenaissSource`, and writes all 6 `public/api/*.json`
   files (overview, categories, health, categories/{id}, index/{id}). Reusing the exact
   handler compute path guarantees the shapes match what the frontend fetches.
   - **Gated:** fetches live only when `--live` / `USE_RENAISS=1` / `VERCEL_ENV=production`
     / partner creds. Otherwise it no-ops and keeps the committed snapshot — so local
     `pnpm build` never spends the 10/day quota.
   - **All-or-nothing + fail-safe:** only writes when the full set fetched and validated as
     genuinely live (`source==='renaiss'`, non-empty). Any error (rate-limit, outage,
     timeout) → keep the committed snapshot, `exit 0` → the build never breaks.
   - **Prunes** per-category files for games Renaiss no longer lists (auto-tracks the set).
2. `package.json` — `build` = `node scripts/gen-snapshot.mjs && vite build` (Vercel runs
   `pnpm run build`, confirmed from deploy logs). Added `gen:snapshot` for manual refresh.
3. `public/api/*.json` — regenerated now to the live Jul-14 data (the committed fallback
   floor is fresh, not Jul-7).
4. README — document the automated build-time refresh + fail-safe + local skip.

## Freshness envelope (be honest about it)
- Every **prod deploy** ships live data. ✓ (the direct ask: "the build makes sure to have
  live data")
- Between deploys the snapshot is static (can't live-fetch per request on the public tier).
  Truly hands-off "never stale while idle" needs a scheduled rebuild (daily GitHub Action /
  Vercel cron + deploy hook) or partner creds for runtime-live — a standing automation, so
  surface it as a follow-up for the user to approve, don't wire it unprompted.
